# Feedback Principles

Use this when the feedback needs clearer prioritization, specificity, uncertainty handling, or a less overwhelming report structure.

## Use this when...

- The user asks for general feedback but needs an actionable revision path.
- You have many possible comments and need to decide what belongs in the final report.
- The artifact is rough, high-stakes, or relationship-sensitive.

## The job of this topic

Turn critique into a bounded, useful decision aid: what works, what fails, what to fix first, and what can wait.

## Default workflow

1. Name the artifact goal and the standard you are using.
2. Identify strengths that the revision should preserve.
3. Separate blocking issues from major issues, secondary improvements, and optional polish.
4. Translate each concern into an action: move, cut, add evidence, narrow a claim, redesign, clarify, rehearse, verify, or discuss.
5. Mark uncertainty when context is missing.
6. End with a revision sequence, not a pile of comments.

## Decision rules

- Give fewer, sharper comments when the user asks for quick feedback.
- Put feasibility, compliance, evidence, and audience failures before style polish.
- Preserve the artifact's useful core before recommending large restructuring.
- Make line edits only after the review mode permits line-level work.
- Use examples or replacement wording when a comment would otherwise be abstract.
- State assumptions instead of silently judging unavailable context.

## Variants and edge cases

- For early drafts, prioritize structure, claim, audience, and missing evidence over wording.
- For near-final artifacts, prioritize readiness blockers, consistency, final-size checks, and submission constraints.
- For mentor-style feedback, include discussion questions and preserve agency.
- For reviewer-style feedback, organize by major/minor concerns and tie concerns to evidence or claims.

## Anti-patterns

- "Looks good" without naming why.
- "Needs clarity" without naming the reader confusion.
- Long unsorted lists of comments.
- Polishing sentences while the artifact has a strategy or evidence problem.
- Giving feedback that the user cannot act on before their deadline.

## Diagnostic questions

- What would make this artifact fail for its intended audience?
- Which comments should the user address before any others?
- Which strengths should not be broken during revision?
- What context is missing enough to change the critique?
- Is the requested feedback mode compatible with the level of detail being returned?

## Output patterns / mini-templates

```text
Goal I am reviewing against: <goal>.
Preserve: <strengths>.
Must fix first: <issue -> action -> rationale>.
Next layer: <issue -> action -> rationale>.
Optional polish: <only if useful>.
Assumptions or missing context: <limits>.
```

## Examples

- Weak: "The introduction is confusing." Strong: "The introduction has three problem statements; choose one and make the rest supporting context."
- Weak: "Make the poster cleaner." Strong: "Cut the methods paragraph by half and use that space to enlarge the result figure the audience must read first."

## When not to apply this

Do not replace specialist domain guidance with generic feedback principles when a domain lens has a concrete rule for the artifact.
