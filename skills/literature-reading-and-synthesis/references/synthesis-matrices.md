# Synthesis Matrices

Use this when the user needs to compare papers, build literature-review notes, map a field, or turn reading into next actions.

## Use this when...

- The user asks what fields belong in a synthesis matrix.
- Several papers need comparison.
- Reading notes need to support a review, presentation, or project decision.
- The user wants to connect papers to a field map or future work.

## The job of this topic

Turn paper notes into comparable, reusable rows. A synthesis matrix should keep claims, evidence, methods, limitations, source context, and next actions separate.

## Default workflow

1. Define the comparison goal: methods, claims, field positions, evidence strength, or project relevance.
2. Choose row units: one paper, one method, one figure, one claim, or one source stream item.
3. Add core columns:
   - paper
   - reading goal
   - article type
   - question or motivation
   - approach or method
   - key result or figure
   - author interpretation
   - limitations or alternatives
   - next steps
   - relationship to other sources
   - impact on current project
4. Add tracking context when useful: source/feed, field layer, query/topic, and action taken.
5. End each row with a build-on-it note.

## Decision rules

- If comparing methods papers, add inputs, assumptions, validation, benchmarks, and failure modes.
- If comparing related claims, add evidence object, claim strength, limitation, and competing interpretation.
- If mapping a field, add source layer, review source, specialty venue, method/tool stream, and known gap.
- If preparing a presentation, add figure IDs, discussion prompts, and unresolved questions.
- If the matrix is becoming a dumping ground, split claim/evidence fields from general notes.

## Variants and edge cases

- **Rapid review:** Use fewer columns but keep claim, evidence, limitation, and next action separate.
- **Project decision:** Include "what this changes" and "decision supported" columns.
- **Field monitoring:** Include source/feed and triage status columns.
- **Dense figure comparison:** Use one row per figure or evidence unit rather than one row per paper.
- **New field:** Use broader field-position and vocabulary columns before detailed method columns.

## Anti-patterns

- One large notes column.
- Mixing result and interpretation.
- Omitting limitations because the table is "just for summary."
- Forgetting why a paper entered the matrix.
- Ending without a next action.

## Diagnostic questions

- What decision or synthesis will the matrix support?
- Are row units consistent?
- Can two papers be compared across the same fields?
- Are source/feed and field-layer context needed?
- Does each row separate claim, evidence, interpretation, and caveat?
- Does each row say what to do next?

## Output patterns / mini-templates

### Paper synthesis row

| Paper | Goal | Type | Question | Approach | Key evidence | Interpretation | Limitations | Related sources | Next action |
|---|---|---|---|---|---|---|---|---|---|

### Field source map

| Layer | Source type | Purpose | Volume/noise | What to extract | Gap |
|---|---|---|---|---|---|

### Build-on-it note

| What this changes | Related sources | New question | Method/design implication | Discussion prompt | Next action |
|---|---|---|---|---|---|

## Examples

- A methods comparison matrix should include validation and failure modes, not only conclusions.
- A field map can represent broad venues, specialty venues, review sources, method/tool sources, author streams, and informal discovery channels.
- A project-relevance matrix should record whether a paper changes the user's next experiment, search, or framing.

## When not to apply this

- Do not build a full matrix when the user only needs a quick paper-readiness checklist.
- Do not use a source-feed map as a current journal recommendation list.
