# Talk Structure

Use this when the user needs a clearer scientific talk arc, conference presentation outline, seminar flow, or slide order.

## The job of this topic

Make the audience understand why the work matters, what claim the talk is proving, and how each section supports that claim. A talk is not a compressed paper; it is a guided live argument under time and attention limits.

## Default workflow

1. Name the audience, talk length, venue, and expected expertise.
2. Write the one-sentence premise before changing slide order.
3. Choose an opening that creates need: problem, gap, decision, surprising observation, or practical consequence.
4. Limit the body to two or three main moves. Each move should answer one audience question.
5. Put methods only where they are needed to make a result credible.
6. Use transitions that tell the audience why the next section follows from the last one.
7. End by restating the specific message, its implication, and what question or action comes next.
8. Keep backup details for anticipated expert questions rather than loading the main path.

## Decision rules

- If the talk reads like a paper outline, rebuild around `problem -> premise -> evidence -> implication`.
- If the audience is mixed, give non-specialists an orientation bridge before specialist detail.
- If the talk has more than three major results, choose the result that best proves the premise and demote the rest.
- If a method is technically impressive but not needed for trust in the claim, move it to backup.
- If the opening starts with institutional history or a full literature tour, replace it with the audience's need to care.
- If timing is tight, cut breadth before cutting orientation.
- If the audience needs a briefing or decision-oriented update, state the governing message first and group support into a few distinct reasons.

## Variants and edge cases

- Short conference talk: one premise, one or two result moves, minimal background, direct close.
- Department seminar: more context is acceptable, but the premise still needs to arrive early.
- Thesis or committee talk: include enough method and future direction to show command, but keep a clear live path.
- Public or cross-disciplinary talk: lead with relevance, plain-language stakes, and concrete examples before mechanism.
- Technical internal talk: keep vocabulary precise, but do not skip the claim or reading path.
- Answer-first briefing: lead with the conclusion when the audience expects a concise update or decision support; avoid this when the point is discovery-style teaching or walking through genuine uncertainty.

## Anti-patterns

- Starting with a table of contents that names sections but not the argument.
- Treating every experiment as equally important.
- Letting caveats arrive before the audience understands the main result.
- Explaining figures before telling the audience what to look for.
- Ending with a vague "future work" slide instead of a specific implication.

## Diagnostic questions

- What is the one sentence the audience should remember?
- What question does each section answer?
- Where does the audience first learn why the problem matters?
- Which slides would disappear if the time were cut by one third?
- Where might a non-specialist get lost, and what bridge fixes that?
- Does the close tell the audience what changed because of the work?

## Output patterns / mini-templates


Slide-order review:

```text

Cut:
Move earlier:
Move later:
Split:
Backup:
```

## Examples

- For a 12-minute conference talk with four result sections, keep the two results that prove the premise and turn the others into backup or one synthesis slide.
- For a cross-disciplinary seminar, open with the biological or practical gap, then define only the technical terms needed for the first result.
- For a committee talk, separate "what we know now" from "what I will test next" so progress and plan are both visible.
- For a decision-oriented project update, put the governing message on the first slide, then organize support as two or three distinct reasons with backup detail available for questions.

## When not to apply this

Do not use this as the main lens when the user only needs manuscript prose, publication figure design, or peer-review strategy.
