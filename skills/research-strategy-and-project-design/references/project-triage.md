# Project Triage

## Use This When

Use this playbook when the user has a fresh idea, an overgrown project, a disliked commitment, or a portfolio of possible directions that needs prioritization.

## The Job Of This Topic

Convert enthusiasm, obligation, and uncertainty into a decision process. The output should identify what to pursue, pause, pivot, or quit, and why.

## Default workflow

1. Let fresh excitement cool enough to compare the idea against alternatives.
2. Screen for impact, feasibility, timeline, technology risk, and fit with the user's role.
3. Name the decisive uncertainty: the one thing that most changes the pursue/drop decision.
4. If the project is already running, set a dated stop/go review rather than letting inertia decide.
5. For evolving projects, log possible branches and deliberately select which to pursue or set aside.

## Decision rules

- Continue only if the future value of the project beats alternative uses of future time.
- Imagine the paper, dataset, tool, or result the project is becoming; if that output would not be worth the remaining work, stop or reshape.
- Separate project evolution from project sprawl: new branches are useful only when selected against goals and constraints.
- Balance the portfolio, not just individual ideas: include a mix of traction, learning, risk, and high-upside work.

## Variants And Edge Cases

- A starter project can be useful even if it is not the long-term direction, but only if its time box is explicit.
- A project with negative early evidence may still be worth a cheap rescue test if the upside is high and the failure mode is specific.
- Collaborator opportunities can open branches, but they also create coordination and ownership costs.

## Anti-Patterns

- Continuing because the user has already invested heavily.
- Continuing because quitting feels like a character flaw.
- Accepting low-cost advice from people who will not pay the time cost.
- Pursuing every branch opened by new data or collaborators.

## Diagnostic Questions

- What decision must be made now: pursue, narrow, pivot, pause, or quit?
- What would the final output look like if the project succeeds?
- What is the cheapest test that could make the project no longer worth doing?
- What alternative project would use the same time and resources?
- Which branches are being kept because they are promising, and which because no one chose to close them?

## Output Patterns / Mini-Templates

For a running project, use:

```text
Current commitment:
Best plausible output:
Remaining cost:
Main uncertainty:
Stop/go review date:
Continue if:
Stop or pivot if:
Next decisive action:
```

## Examples

- If a project depends on a technology that may not work, test that dependency before expanding biological scope.
- If the user dislikes a starter project and cannot name a worthwhile output, recommend a short stop/go review and a concrete alternative.
- If many branches emerged from early data, ask the user to rank branches by expected learning and cost before adding experiments.

## When Not To Apply This

Do not use project triage to dismiss obligations that have ethical, supervisory, contractual, or collaboration commitments without separately addressing those constraints.
