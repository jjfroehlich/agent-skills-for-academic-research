# Choosing Problems

## Use This When

Use this playbook when the user is deciding which scientific problem is worth pursuing, assigning projects across career stages, or asking how to balance novelty, payoff, and tractability.

## The Job Of This Topic

Make problem choice explicit. A good answer should distinguish the value of the question from the feasibility of answering it, then show why one option is better than nearby alternatives.

## Default workflow

1. List the candidate problems and the decision horizon.
2. Score each problem for expected knowledge gain and feasibility with the user's actual tools, data, collaborators, and time.
3. Remove dominated options: ideas that are both less feasible and less informative than another candidate.
4. For the remaining options, check stage fit, portfolio role, and whether the project can be broken into smaller decisive steps.
5. Recommend a problem, a narrowed version of it, or a short exploration that would make the choice clearer.

## Decision rules

- Prefer projects near the feasibility-interest frontier: high expected learning for the available effort.
- Match risk to stage. Beginners usually need traction and confidence; later trainees can move toward higher-interest risk; independent labs can split grand challenges into smaller projects.
- For high-novelty ideas, name the assumption, bottleneck, standard practice, or expected pattern the project would change.
- Search the adjacent possible when a field feels saturated: nearby methods, concepts, or organisms can reveal questions that are close enough to transfer but far enough to be non-obvious.
- Treat not-knowing as productive only after baseline reading and feasibility checks show the uncertainty is tied to a meaningful unanswered question.

## Variants And Edge Cases

- For disruptive projects, pair the creative question with staged evidence that could later persuade skeptics.
- For interdisciplinary projects, check transfer cost as well as novelty; distant analogies can be generative but expensive.
- For trainee projects, include independence, publication horizon, and skill development as feasibility constraints.

## Anti-Patterns

- Choosing the hardest problem because it sounds most important.
- Choosing the easiest problem because it is already set up.
- Confusing a vague gap in the literature with a tractable research question.
- Treating discomfort as proof of depth when the missing piece is ordinary background preparation.

## Diagnostic Questions

- What would be learned if this project worked?
- What concrete observation would make the question more or less interesting?
- Which candidate is dominated by another on both feasibility and expected knowledge gain?
- What assumption or practice could this project revise?
- Is the project a good fit for the user's stage, time horizon, and available support?

## Output Patterns / Mini-Templates

Use this compact comparison when there are multiple options:

| Candidate | Knowledge gain | Feasibility | Stage fit | Main uncertainty | Recommendation |
|---|---|---|---|---|---|
| Option A | High/medium/low | High/medium/low | Good/risky/poor | What must be true? | Pursue, narrow, test, or drop |

## Examples

- If an idea is high-interest but low-feasibility, recommend a smaller entry point or collaborator before committing.
- If an idea is feasible but low-interest, use it only as training, infrastructure, or a fast support project.
- If a field feels predictable, ask what assumption everyone is preserving because it makes experiments easier.

## When Not To Apply This

Do not use problem-choice triage as a substitute for ethics review, safety review, field-specific technical feasibility, or statistical design.
