# Budget And Feasibility

## Use this when...

Use this when the proposal needs stronger budget logic, preliminary data selection, feasibility evidence, risk plans, applicant-capacity evidence, or project-deliverability review.

## The job of this topic

Show reviewers that the proposed work can be executed with the requested resources, available evidence, applicant capacity, and credible alternatives.

## Default workflow

1. List promised work packages and the resources each requires.
2. Check personnel, equipment, travel, services, data, institutional support, and funder-allowed costs.
3. Select preliminary data because it answers reviewer feasibility concerns, not because it shows general productivity.
4. Pair each risky component with feasibility evidence, interpretable outcomes, and a backup.
5. Use CV, publication, engagement, independence, environment, or team evidence to prove execution capacity.
6. For future-work grants, remove completed work from the funded plan or recast it as preliminary evidence.

## Decision rules

- If the budget cannot deliver the promised work, rescope the project or revise the costs.
- If preliminary data do not support a feasibility claim, replace or reorganize them.
- If execution depends on unavailable data, undeveloped methods, or uncertain access, add evidence or change scope.
- If work is already complete, do not present it as proposed future work.

## Variants and edge cases

- Fellowships often need applicant, lab, mentor, and environment evidence.
- Early-career grants may need independence evidence without turning the application into a CV prize.
- Empirical projects may need access, representativeness, mergeability, and identification strategy.
- Some project problems after award should be handled through funder or research-office communication rather than silence.

## Anti-patterns

- Under-budgeting to look efficient while making the project undeliverable.
- Showing random preliminary data as a productivity record.
- Hiding missing data access or method development behind "high risk" language.
- Packaging completed or circulating work as future funded work.

## Diagnostic questions

- What reviewer concern does each preliminary data item reduce?
- Which work package is least deliverable with the requested resources?
- Which costs are disallowed or under-supported?
- What evidence proves the applicant or team can execute the plan?
- What is the fallback if the main method, data source, or recruitment path fails?

## Output patterns / mini-templates

Feasibility evidence map:


## Examples

Weak: "Preliminary data show the lab has worked on this topic."

Stronger: "Preliminary data show that the proposed assay detects the target signal at the required resolution, reducing the main feasibility risk in Aim 1."

## When not to apply this

Do not provide institutional finance or legal advice. Flag budget-rule uncertainty and tell the user what must be checked with the funder or institution.
